# Migration Guide: `cors_origins` → `cors_allowed_origins` Rename

**Date:** 2026-06-30
**Affects:** 26 files across backend, infrastructure, and test layers

---

## Summary

The `Settings` model field `cors_origins` has been renamed to `cors_allowed_origins`. This is a purely cosmetic rename to improve naming consistency, but because Pydantic derives environment variable names directly from field names, this change also renames the required environment variable from `CORS_ORIGINS` to `CORS_ALLOWED_ORIGINS`. Any code, configuration file, Docker definition, or test that references the old field name or environment variable **will break**. The most severe consequence is an `AttributeError` at application startup that crashes the server entirely. All affected deployments must be updated before or alongside deploying the new backend code.

---

## Breaking Changes

### 1. `Settings` field renamed

The `cors_origins` field no longer exists on the `Settings` model.

**Before:**
```python
# backend/app/config.py (or equivalent)
class Settings(BaseSettings):
    cors_origins: str = "http://localhost:5173"
```

**After:**
```python
class Settings(BaseSettings):
    cors_allowed_origins: str = "http://localhost:5173"
```

---

### 2. Environment variable renamed

Pydantic resolves environment variables by uppercasing the field name. The expected environment variable has therefore changed.

| | Old | New |
|---|---|---|
| **Env var name** | `CORS_ORIGINS` | `CORS_ALLOWED_ORIGINS` |
| **Settings field** | `settings.cors_origins` | `settings.cors_allowed_origins` |

> ⚠️ **Silent failure risk:** If `CORS_ORIGINS` is still set but `CORS_ALLOWED_ORIGINS` is not, Pydantic will silently fall back to the default value `http://localhost:5173`. There is **no warning or error** — CORS will simply be misconfigured in non-localhost environments.

---

### 3. `CORSMiddleware` configuration in `main.py` crashes at startup

**Before:**
```python
# backend/app/main.py
app.add_middleware(
    CORSMiddleware,
    allow_origins=[settings.cors_origins],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

**After:**
```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=[settings.cors_allowed_origins],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

> 🔴 **Severity: CRITICAL** — Failure to make this change causes an `AttributeError` at startup, taking the entire server down.

---

### 4. `.env.example` documents the wrong variable name

**Before:**
```dotenv

```

**After:**
```dotenv
# backend/.env.example
CORS_ALLOWED_ORIGINS=http://localhost:5173
```

**Before:**

**After:**

---

**Before:**
```dockerfile

```

**After:**
```dockerfile

```

---

### 7. Test fixtures referencing the old field name fail at collection time

**Before:**
```python
# backend/tests/conftest.py
@pytest.fixture
def settings_override():
    return Settings(cors_origins="http://testserver")

# — or —
settings.cors_origins = "http://testserver"
```

**After:**
```python
@pytest.fixture
def settings_override():
    return Settings(cors_allowed_origins="http://testserver")

# — or —
settings.cors_allowed_origins = "http://testserver"
```

> 🔴 **Severity: HIGH** — Pydantic raises a `ValidationError` or `AttributeError` at fixture construction, causing **all tests that depend on this fixture to fail at collection time**, not just at runtime.

## Required Actions by Role

### 🔧 Backend Developer

1. **`backend/app/main.py`** — Replace every occurrence of `settings.cors_origins` with `settings.cors_allowed_origins` in the `CORSMiddleware` setup. *(CRITICAL)*
2. **`backend/app/common/security.py`** — Search for any origin validation logic that reads `settings.cors_origins` and update to `settings.cors_allowed_origins`.
3. **`backend/app/common/deps.py`** — Same search-and-replace as above.
4. **`backend/.env.example`** — Rename the documented variable from `CORS_ORIGINS` to `CORS_ALLOWED_ORIGINS`. *(HIGH)*
5. **Any other backend file** — Run a full codebase search (see [Code Examples](#code-examples)) to catch any remaining references.
6. **Coordinate with DevOps** to ensure all deployment environment variables are updated before or simultaneously with the code deployment.

---

### 🐳 DevOps / Infrastructure

1. **`docker-compose.yml`** — Rename `CORS_ORIGINS` to `CORS_ALLOWED_ORIGINS` in all service `environment` blocks. *(HIGH)*
2. **`backend/Dockerfile`** — Update any `ENV CORS_ORIGINS=...` directives to `ENV CORS_ALLOWED_ORIGINS=...`. *(MEDIUM)*
3. **CI/CD pipelines** — Update any pipeline environment variable definitions (e.g., GitHub Actions secrets/env blocks, GitLab CI variables, Kubernetes ConfigMaps/Secrets) that set `CORS_ORIGINS`.
4. **Production/staging `.env` files** — Rename the key in all deployed environment files.
5. **Verify** there is no period where the new code is live but environment variables have not yet been updated — this will cause the server to start with `http://localhost:5173` as the only allowed origin.

---

### 🧪 QA / Test Engineer

1. **`backend/tests/conftest.py`** — Update all `Settings(cors_origins=...)` constructor calls and `settings.cors_origins = ...` assignments to use `cors_allowed_origins`. *(HIGH)*
2. **All test helpers and factories** — Search for any other test file that constructs `Settings` with `cors_origins` as a keyword argument.
3. **Run the full test suite** after changes and confirm zero collection errors related to `Settings`.
4. **Add or update CORS integration tests** to assert that:
   - The allowed origin is correctly read from `CORS_ALLOWED_ORIGINS`.
   - Requests from `http://localhost:5173` are accepted (default).
   - Requests from an unlisted origin are rejected with HTTP `403`.
5. **Verify the old variable name `CORS_ORIGINS` is no longer accepted** — set only `CORS_ORIGINS` in a test environment and confirm CORS behaves as if unconfigured (i.e., falls back to default).
