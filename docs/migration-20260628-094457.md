# Migration Guide: `final_amount` Field Added to `OrderOut`

**Date:** 2026-06-28
**Severity:** CRITICAL / HIGH
**Affected Teams:** Backend, Frontend, QA

---

## Summary

A new non-nullable integer field, `final_amount`, has been added to the `OrderOut` schema and the underlying `Order` database model. This field is **required** — it has no default value and cannot be `null` — which means every layer of the stack must be updated in lockstep: the SQLAlchemy model, the Pydantic schema, the order service logic, the router handlers, the seed script, and the frontend TypeScript types and components. Failure to update any single layer will result in runtime crashes, Pydantic `ValidationError` exceptions, database `IntegrityError`s, or silent `undefined` rendering in the UI. All five impacted files must be addressed before this change is deployed to any shared environment.

---

## Breaking Changes

### 1. `OrderOut` Pydantic Schema — Missing Required Field

The `OrderOut` schema no longer validates without `final_amount`. Any endpoint that constructs or returns an `OrderOut` object without this field will raise a `ValidationError`.

**Before:**
```python
# backend/app/schemas/order.py
class OrderOut(BaseModel):
    id: int
    user_id: int
    status: str
    created_at: datetime
```

**After:**
```python
# backend/app/schemas/order.py
class OrderOut(BaseModel):
    id: int
    user_id: int
    status: str
    created_at: datetime
    final_amount: int  # ← REQUIRED, non-nullable
```

---

### 2. SQLAlchemy `Order` Model — Missing Column Definition

Without the column on the ORM model, accessing `order.final_amount` raises an `AttributeError` and Alembic will report migration inconsistencies.

**Before:**
```python
# backend/app/models/order.py
class Order(Base):
    __tablename__ = "orders"

    id = Column(Integer, primary_key=True)
    user_id = Column(Integer, ForeignKey("users.id"), nullable=False)
    status = Column(String, nullable=False)
    created_at = Column(DateTime, default=datetime.utcnow)
```

**After:**
```python
# backend/app/models/order.py
class Order(Base):
    __tablename__ = "orders"

    id = Column(Integer, primary_key=True)
    user_id = Column(Integer, ForeignKey("users.id"), nullable=False)
    status = Column(String, nullable=False)
    created_at = Column(DateTime, default=datetime.utcnow)
    final_amount = Column(Integer, nullable=False)  # ← ADD THIS
```

---

### 3. Order Service — `final_amount` Must Be Computed and Assigned

Every code path in `order_service.py` that creates or updates an order must populate `final_amount` before returning. Unset fields cause Pydantic to raise `ValidationError` on every affected endpoint.

**Before:**
```python
# backend/app/services/order_service.py
def create_order(db: Session, order_data: OrderCreate) -> OrderOut:
    order = Order(
        user_id=order_data.user_id,
        status="pending",
    )
    db.add(order)
    db.commit()
    db.refresh(order)
    return OrderOut.from_orm(order)
```

**After:**
```python
# backend/app/services/order_service.py
def create_order(db: Session, order_data: OrderCreate) -> OrderOut:
    final_amount = _calculate_final_amount(order_data)  # ← COMPUTE VALUE

    order = Order(
        user_id=order_data.user_id,
        status="pending",
        final_amount=final_amount,  # ← ASSIGN FIELD
    )
    db.add(order)
    db.commit()
    db.refresh(order)
    return OrderOut.from_orm(order)

def _calculate_final_amount(order_data: OrderCreate) -> int:
    # Implement your business logic here.
    # Example: sum of item prices after discounts, in cents.
    return sum(item.unit_price * item.quantity for item in order_data.items)
```

> ⚠️ **Note:** Apply the same change to any `update_order` or `patch_order` service methods that return `OrderOut`.

---

### 4. Seed Script — Orders Must Include `final_amount`

Because the column is `nullable=False` with no database-level default, inserting seed records without `final_amount` will raise a database `IntegrityError`.

**Before:**
```python
# backend/app/seed.py
def seed_orders(db: Session):
    orders = [
        Order(user_id=1, status="completed"),
        Order(user_id=2, status="pending"),
    ]
    db.add_all(orders)
    db.commit()
```

**After:**
```python
# backend/app/seed.py
def seed_orders(db: Session):
    orders = [
        Order(user_id=1, status="completed", final_amount=4999),  # ← ADD FIELD
        Order(user_id=2, status="pending",   final_amount=1250),  # ← ADD FIELD
    ]
    db.add_all(orders)
    db.commit()
```

### 6. Frontend Component — Stale Query Cache May Return `undefined`

TanStack Query may serve cached responses that predate this API change. Derived calculations will silently produce `NaN` or `0` unless the cache is invalidated.

**Before:**
```tsx
// frontend/src/pages/OrdersPage.tsx
function OrdersPage() {
  const { data: orders } = useOrders();

  return (
    <table>
      {orders?.map(order => (
        <tr key={order.id}>
          <td>{order.status}</td>
          {/* final_amount not rendered — field did not exist */}
        </tr>
      ))}
    </table>
  );
}
```

**After:**
```tsx
// frontend/src/pages/OrdersPage.tsx
function OrdersPage() {
  const queryClient = useQueryClient();
  const { data: orders } = useOrders();

  // Invalidate stale order cache on mount to ensure fresh data.
  useEffect(() => {
    queryClient.invalidateQueries({ queryKey: ["orders"] });
  }, [queryClient]);

  return (
    <table>
      <thead>
        <tr>
          <th>Status</th>
          <th>Final Amount</th>
        </tr>
      </thead>
      <tbody>
        {orders?.map(order => (
          <tr key={order.id}>
            <td>{order.status}</td>
            <td>${(order.finalAmount / 100).toFixed(2)}</td> {/* ← RENDER FIELD */}
          </tr>
        ))}
      </tbody>
    </table>
  );
}
```

## Required Actions by Role

### 🔧 Backend Developer

1. **`backend/app/models/order.py`** — Add `final_amount = Column(Integer, nullable=False)` to the `Order` model.
2. **`backend/app/schemas/order.py`** — Add `final_amount: int` to the `OrderOut` Pydantic schema as a required field.
3. **`backend/app/services/order_service.py`** — Implement `final_amount` calculation logic and assign it during order creation and all update operations.
4. **`backend/app/routers/orders.py`** — Verify that all route handlers (GET, POST, PATCH) that return `OrderOut` will now include `final_amount`. No direct construction of `OrderOut` should omit the field.
5. **`backend/app/seed.py`** — Add a realistic `final_amount` integer value to every seeded `Order` record.
6. **Database Migration** — Generate and apply an Alembic migration for the new non-nullable column. If the table already has rows, provide a default value in the migration script before removing the server default.
   ```bash
   alembic revision --autogenerate -m "add_final_amount_to_orders"
   alembic upgrade head
   ```

---

### 🖥️ Frontend Developer

1. **TypeScript Interface** — Add `finalAmount: number` to the `Order` interface in your types file or `OrdersPage.tsx`.
2. **`frontend/src/pages/OrdersPage.tsx`** — Update JSX to render `order.finalAmount` wherever order totals are displayed.
3. **TanStack Query Cache** — Invalidate the `"orders"` query key on component mount (or bump the query key version) to prevent stale pre-migration data from being served.
4. **Derived Calculations** — Audit any hooks or utility functions that compute totals or summaries from order data. Ensure they use `finalAmount` rather than a previously derived workaround, and add guards against `undefined` during the cache transition period.
5. **API Client / Response Mapper** — If your API client maps snake_case to camelCase, confirm `final_amount` → `finalAmount` conversion is handled correctly.

---

### 🧪 QA Engineer

1. **API Contract Testing** — Verify that all order endpoints (`GET /orders`, `GET /orders/{id}`, `POST /orders`, `PATCH /orders/{id}`) return a response body containing a non-null integer `final_amount` field.
2. **Negative Testing** — Confirm that attempting to create an order without `final_amount` (via direct API call bypassing the service) returns a `422 Unprocessable Entity`, not a `500`.
3. **Seed Data Validation** — Run the seed script in a clean environment and confirm it completes without `IntegrityError`. Query the database to verify all seeded orders have a non-null `final_amount`.
4. **Frontend Regression** — Load the Orders page with a cleared browser cache and confirm `finalAmount` renders correctly in the UI without `undefined` or `NaN` appearing.
5. **Stale Cache Scenario** — Simulate a user session that was active before the deployment (by manually injecting old cached data into TanStack Query's cache), then navigate to `OrdersPage` and confirm the cache invalidation logic fetches fresh data.
6. **Database Migration** — Validate the Alembic migration applies cleanly on a copy of the production database, including backfill of any existing rows.

```python
# backend/app/services/order_service.py
from sqlalchemy.orm import Session
from backend.app.models.order import Order
from backend.app.schemas.order import OrderCreate, OrderOut

def create_order(db: Session, order_data: OrderCreate) -> OrderOut:
    final_amount = _calculate_final_amount(order_data)

    order = Order(
        user_id=order_data.user_id,
        status="pending",
        final_amount=final_amount,
    )
    db.add(order)
    db.commit()
    db.refresh(order)
    return OrderOut.from_orm(order)

def _calculate_final_amount(order_data: OrderCreate) -> int:
    """Returns the final order amount in the smallest currency unit (e.g., cents)."""
    subtotal = sum(item.unit_price * item.quantity for item in order_data.items)
    discount = order_data.discount_amount or 0
    return max(subtotal - discount, 0)
```

### Alembic Migration for Existing Data

```python
# alembic/versions/xxxx_add_final_amount_to_orders.py
from alembic import op
import sqlalchemy as sa

def upgrade():
    # Step 1: Add column as nullable to avoid IntegrityError on existing rows.
    op.add_column("orders", sa.Column("final_amount", sa.Integer(), nullable=True))

    # Step 2: Backfill existing rows with a sensible default (adjust logic as needed).
    op.execute("UPDATE orders SET final_amount = 0 WHERE final_amount IS NULL")

    # Step 3: Enforce NOT NULL constraint now that all rows have a value.
    op.alter_column("orders", "final_amount", nullable=False)

def downgrade():
    op.drop_column("orders", "final_amount")
```

> ⚠️ **Do not** add the column as `nullable=False` directly if the table already contains rows. Always backfill first.
