# Migration Guide: `ProductStatus` Enum — Addition of `LOW_STOCK` Value

**Date:** 2026-06-29
**Severity:** HIGH (action required before next deployment)
**Affected Files:** 14

---

## Summary

The `ProductStatus` enum has been extended with a new value: `LOW_STOCK`. This value will now be returned by the backend API when a product's inventory falls below a defined threshold, and it may also be sent as a filter parameter in API requests. Because this enum is shared across the backend (Python/Pydantic), the frontend (TypeScript), and the test suite, **every layer of the stack must be updated before this change is deployed**. Failure to update any layer will result in HTTP 422/500 errors from the API, broken product listings in the UI, and silent type failures in TypeScript code. This guide details every required change, organized by role, with before/after examples and a deployment checklist.

---

## Breaking Changes

### 1. `ProductStatus` Enum — New Value Added

| | Before | After |
|---|---|---|
| **Values** | `ACTIVE`, `ARCHIVED`, `DRAFT`, `SOLD_OUT` | `ACTIVE`, `ARCHIVED`, `DRAFT`, `LOW_STOCK`, `SOLD_OUT` |

Any code that exhaustively handles `ProductStatus` values (switch statements, if/elif chains, Pydantic validators, TypeScript discriminated unions) will break or silently misbehave when it encounters `LOW_STOCK`.

---

### 2. Backend Pydantic Schema Rejects `LOW_STOCK`

**Before** — `backend/app/schemas/product.py`:
```python
class ProductStatus(str, Enum):
    ACTIVE = "ACTIVE"
    ARCHIVED = "ARCHIVED"
    DRAFT = "DRAFT"
    SOLD_OUT = "SOLD_OUT"
```

**After**:
```python
class ProductStatus(str, Enum):
    ACTIVE = "ACTIVE"
    ARCHIVED = "ARCHIVED"
    DRAFT = "DRAFT"
    LOW_STOCK = "LOW_STOCK"   # ← NEW
    SOLD_OUT = "SOLD_OUT"
```

> **Impact:** Without this change, any API request or response containing `status: "LOW_STOCK"` raises a Pydantic `ValidationError`, returning HTTP 422 or 500 to the client.

---

### 5. Order Service Does Not Handle `LOW_STOCK`

**Before** — `backend/app/services/order_service.py`:
```python
def can_place_order(product: Product) -> bool:
    if product.status == ProductStatus.SOLD_OUT:
        raise OrderError("Product is sold out.")
    # LOW_STOCK is unhandled — silently treated as ACTIVE
    return True
```

**After**:
```python
def can_place_order(product: Product) -> bool:
    if product.status == ProductStatus.SOLD_OUT:
        raise OrderError("Product is sold out.")
    if product.status == ProductStatus.LOW_STOCK:
        # Allow order but surface a warning to the caller
        logger.warning(
            "Order placed for low-stock product: %s", product.id
        )
    return True
```

> **Note:** The exact business logic (allow, warn, or block) for `LOW_STOCK` orders must be confirmed with the product team before implementation. The example above **allows** the order with a warning. Adjust to match your requirements.

## Required Actions by Role

2. **`backend/app/models/__init__.py`** — If `ProductStatus` is redefined or re-exported here, update it to include `LOW_STOCK`.
3. **`backend/app/routers/products.py`** — Verify that status filter query parameters and status update endpoints accept `LOW_STOCK`. Update any `Enum` references or explicit value lists.
4. **`backend/app/routers/orders.py`** — Check whether order endpoints inspect or return `ProductStatus`. Ensure `LOW_STOCK` is handled correctly.
5. **`backend/app/services/order_service.py`** — Implement explicit business logic for `LOW_STOCK` products (see Code Examples above). Confirm with product team whether orders should be allowed, warned, or blocked.

7. **`backend/app/main.py`** — If `ProductStatus` values are referenced directly (e.g., in OpenAPI docs customization, middleware, or startup checks), add `LOW_STOCK`.

```typescript
// BEFORE — missing LOW_STOCK; TypeScript won't catch it without a never guard
function getStatusBadge(status: ProductStatus): JSX.Element {

    case "ACTIVE":   return <Badge color="green">Active</Badge>;
    case "ARCHIVED": return <Badge color="gray">Archived</Badge>;
    case "DRAFT":    return <Badge color="yellow">Draft</Badge>;
    case "SOLD_OUT": return <Badge color="red">Sold Out</Badge>;
    default:
      return <Badge color="gray">Unknown</Badge>; // LOW_STOCK silently hits here

// AFTER — LOW_STOCK handled; never guard catches future omissions at compile time
function assertNever(value: never): never {
  throw new Error(`Unhandled ProductStatus: ${value}`);
}

function getStatusBadge(status: ProductStatus): JSX.Element {
  switch (status) {
    case "ACTIVE":    return <Badge color="green">Active</Badge>;
    case "ARCHIVED":  return <Badge color="gray">Archived</Badge>;
    case "DRAFT":     return <Badge color="yellow">Draft</Badge>;
    case "LOW_STOCK": return <Badge color="orange">Low Stock</Badge>; // ← NEW
    case "SOLD_OUT":  return <Badge color="red">Sold Out</Badge>;
    default:
      return assertNever(status); // TypeScript error if any case is missing
  }
}
```

> **Recommendation:** Add `assertNever` as a shared utility and adopt it in all exhaustive switches over `ProductStatus`. This converts future enum additions from silent runtime bugs into compile-time errors.
