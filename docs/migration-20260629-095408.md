# Migration Guide: `calculate_discount` Signature Change & VIP Discount Rate Update

**Date:** 2026-06-29
**Severity:** CRITICAL / HIGH
**Affected Files:** 3

---

## Summary

Two breaking changes were introduced to the discount calculation module. First, the `coupon_code` parameter in `calculate_discount()` was moved from a positional argument to a keyword-only argument, meaning any call site that passes `coupon_code` as a third positional argument will raise a `TypeError` at runtime. Second, the VIP discount rate (`GRADE_DISCOUNT_RATE[UserGrade.VIP]`) was raised from `0.10` (10%) to `0.15` (15%), which silently affects all order totals for VIP users and invalidates any test assertions, cached values, or API documentation anchored to the old rate. Both changes require immediate action before deployment; unpatched code will produce runtime crashes or incorrect pricing for VIP customers.

---

## Breaking Changes

### 1. `calculate_discount()` — `coupon_code` is now keyword-only

The `coupon_code` parameter can no longer be passed as a positional argument. Calls that relied on positional order will raise `TypeError`.

| | Signature |
|---|---|
| **Before** | `calculate_discount(subtotal: int, grade: UserGrade, coupon_code: str \| None = None) -> int` |
| **After** | `calculate_discount(subtotal: int, grade: UserGrade, *, coupon_code: str \| None = None) -> int` |

**Before — positional call (now broken):**
```python
# TypeError: calculate_discount() takes 2 positional arguments but 3 were given
result = calculate_discount(5000, UserGrade.GOLD, "SAVE10")
```

**After — keyword-only call (required):**
```python
result = calculate_discount(5000, UserGrade.GOLD, coupon_code="SAVE10")
```

> **Note:** Calls that omit `coupon_code` entirely (`calculate_discount(5000, UserGrade.GOLD)`) are unaffected and continue to work.

---

### 2. `GRADE_DISCOUNT_RATE[UserGrade.VIP]` — Rate increased from 10% to 15%

| | Value |
|---|---|
| **Before** | `0.1` |
| **After** | `0.15` |

Any hardcoded expectations of a 10% VIP discount — in tests, API response contracts, seed data, or discount-cap logic — are now incorrect. This change is silent: no exception is raised, but VIP order totals will be lower than previously calculated.

**Before:**
```python
# Old: VIP user gets 10% off a $200 order → $20 discount
assert calculate_discount(20000, UserGrade.VIP) == 2000
```

**After:**
```python
# New: VIP user gets 15% off a $200 order → $30 discount
assert calculate_discount(20000, UserGrade.VIP) == 3000
```

## Required Actions by Role

### 🔧 Backend Developer

1. **Audit every call site** of `calculate_discount()` in `order_service.py` and `orders.py`. Search the entire codebase for `calculate_discount(` and verify each invocation.
2. **Rewrite all positional `coupon_code` calls** to use explicit keyword syntax (`coupon_code=...`).
3. **Search for hardcoded `0.1` or `10%`** VIP discount multipliers, discount caps, or minimum charge logic in `order_service.py`. Update them to `0.15`.
4. **Check database seed data and migration scripts** for any hardcoded VIP discount values; update accordingly.
5. **Run `mypy`** (or your configured type checker) across `order_service.py` and `orders.py` to catch any residual parameter-kind violations.
6. **Review `orders.py` router responses** — if the `discount_rate` field is serialized in any response model, verify it reflects `0.15` for VIP users.
7. **Check any discount-cap or maximum-discount guard logic** — a higher discount rate may now exceed previously safe thresholds.

### 🖥️ Frontend Developer

1. **Review API documentation and OpenAPI specs** for any endpoint that returns a VIP discount rate or discounted price. The documented value of `0.1` is now stale.
2. **Audit client-side discount display logic** — if the frontend independently calculates or displays a "10% VIP discount" label, update the copy and/or calculation to reflect 15%.
3. **Coordinate with Backend** to confirm whether `discount_rate` is included in API responses. If it is, update any hardcoded frontend constants that compare against or display this value.
4. **Test checkout flows end-to-end** with a VIP account to confirm displayed totals match the new 15% rate.

### 🧪 QA Engineer

1. **Re-run the full test suite** (`backend/tests/test_discount.py`) after patches are applied and confirm zero `TypeError` failures.
2. **Update all test cases in `test_discount.py`** that call `calculate_discount` with a positional `coupon_code` argument.
3. **Update all VIP discount assertions** in `test_discount.py` and any other test files from 10% expected values to 15%.
4. **Add or update regression tests** that explicitly validate the new 15% VIP discount rate.
5. **Perform integration testing** on the orders API (`orders.py` router) with a VIP user to verify the correct discount appears in responses.
6. **Test boundary cases:** VIP user with no coupon, VIP user with a coupon, non-VIP user with a coupon (should be unaffected by the rate change).
7. **Verify that no cached discount values** (e.g., Redis, in-memory caches, database columns) are serving stale 10% discount results.

## Code Examples

### Fix 1: Update `calculate_discount` call sites

**`backend/app/services/order_service.py` — Before:**
```python
from app.module import calculate_discount
from app.models import UserGrade

def create_order(user, subtotal: int, coupon_code: str | None = None) -> dict:
    # BROKEN: coupon_code passed as positional argument
    discount = calculate_discount(subtotal, user.grade, coupon_code)
    final_price = subtotal - discount
    return {"subtotal": subtotal, "discount": discount, "total": final_price}
```

**`backend/app/services/order_service.py` — After:**
```python
from app.module import calculate_discount
from app.models import UserGrade

def create_order(user, subtotal: int, coupon_code: str | None = None) -> dict:
    # FIXED: coupon_code passed as keyword argument
    discount = calculate_discount(subtotal, user.grade, coupon_code=coupon_code)
    final_price = subtotal - discount
    return {"subtotal": subtotal, "discount": discount, "total": final_price}
```

### Fix 2: Update test call sites and VIP assertions

**`backend/tests/test_discount.py` — Before:**
```python
from app.module import calculate_discount
from app.models import UserGrade

def test_discount_with_coupon():
    # BROKEN: TypeError — coupon_code as positional
    result = calculate_discount(10000, UserGrade.GOLD, "WELCOME")
    assert result == 1500

def test_vip_discount_no_coupon():
    # BROKEN: TypeError + wrong expected value
    result = calculate_discount(10000, UserGrade.VIP, None)
    assert result == 1000  # Expected 10% of 10000

def test_vip_discount_with_coupon():
    # BROKEN: both issues — positional arg + stale assertion
    result = calculate_discount(10000, UserGrade.VIP, "VIP20")
    assert result == 1000
```

**`backend/tests/test_discount.py` — After:**
```python
from app.module import calculate_discount
from app.models import UserGrade

def test_discount_with_coupon():
    # FIXED: coupon_code as keyword argument
    result = calculate_discount(10000, UserGrade.GOLD, coupon_code="WELCOME")
    assert result == 1500

def test_vip_discount_no_coupon():
    # FIXED: no coupon_code argument needed (uses default None); updated to 15%
    result = calculate_discount(10000, UserGrade.VIP)
    assert result == 1500  # 15% of 10000

def test_vip_discount_with_coupon():
    # FIXED: keyword argument + updated assertion for 15% VIP rate
    result = calculate_discount(10000, UserGrade.VIP, coupon_code="VIP20")
    assert result == 1500  # Update if coupon stacks; verify with business logic
```

### Fix 3: Catch hardcoded VIP rate references in service or router logic

**`backend/app/services/order_service.py` — Before:**
```python
VIP_DISCOUNT_CAP = 0.10  # Hardcoded to match old constant — now stale

def validate_discount(grade: UserGrade, rate: float) -> bool:
    if grade == UserGrade.VIP and rate > 0.10:
        raise ValueError("VIP discount exceeds maximum allowed rate")
    return True
```

**`backend/app/services/order_service.py` — After:**
```python
from app.constants import GRADE_DISCOUNT_RATE  # Import from source of truth

def validate_discount(grade: UserGrade, rate: float) -> bool:
    max_rate = GRADE_DISCOUNT_RATE[grade]
    if rate > max_rate:
        raise ValueError(f"Discount rate {rate} exceeds maximum {max_rate} for grade {grade}")
    return True
```
