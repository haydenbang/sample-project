# Migration Guide: `Product.brand_id` Field Addition

**Date:** 2026-06-30
**Severity:** HIGH (3 high-severity risks across 23 impacted files)
**Change Type:** Non-breaking DB schema addition, breaking serialization/type contract

---

## Summary

A new nullable integer column, `brand_id`, has been added to the `Product` model (`Mapped[int | None]`). While the database-level change is additive and non-destructive, the field is **not yet propagated** through the Pydantic schemas, TypeScript types, API routers, or service layer — meaning API responses will silently omit `brand_id`, write operations will silently strip it, and TypeScript consumers will have no type-level awareness of its existence. Any brand-related business logic (filtering, pricing, discounts) will evaluate against `None` indefinitely until this migration is completed. All 23 impacted files must be updated before this feature is considered safe to ship.

---

## Breaking Changes

### 1. Pydantic Schema Missing `brand_id` — API Serialization Failure

The `ProductSchema` in `backend/app/schemas/product.py` does not include `brand_id`. This causes two failure modes: validation errors on strict schema configurations, or silent field omission from JSON responses.

**Before (`backend/app/schemas/product.py`):**
```python
class ProductSchema(BaseModel):
    id: int
    name: str
    price: float
    # brand_id absent
```

**After:**
```python
class ProductSchema(BaseModel):
    id: int
    name: str
    price: float
    brand_id: Optional[int] = None  # ← ADD THIS
```

---

### 2. TypeScript `Product` Type Missing `brand_id` — Silent Runtime Failures

The `Product` type in `frontend/src/types/product.ts` does not declare `brand_id`. TypeScript will not raise a compile error when the field is accessed, but the value will always be `undefined`.

**Before (`frontend/src/types/product.ts`):**
```typescript
export interface Product {
  id: number;
  name: string;
  price: number;
  // brand_id absent
}
```

**After:**
```typescript
export interface Product {
  id: number;
  name: string;
  price: number;
  brand_id?: number | null;  // ← ADD THIS
}
```

---

### 3. Router Not Accepting/Returning `brand_id` — Write Operations Silently Strip Field

Even if the client sends `brand_id` in a POST or PUT request body, the router will strip it if the request schema doesn't include the field, meaning the value is never persisted.

**Before (`backend/app/routers/products.py`):**
```python
@router.post("/products", response_model=ProductSchema)
async def create_product(product: ProductCreateSchema, db: Session = Depends(get_db)):
    db_product = Product(name=product.name, price=product.price)
    # brand_id never set
    ...
```

**After:**
```python
@router.post("/products", response_model=ProductSchema)
async def create_product(product: ProductCreateSchema, db: Session = Depends(get_db)):
    db_product = Product(
        name=product.name,
        price=product.price,
        brand_id=product.brand_id,  # ← ADD THIS
    )
    ...
```

---

### 4. Order Schema Embeds Incomplete `ProductSchema` — Order Responses Omit Brand

`backend/app/schemas/order.py` references `ProductSchema`. Once `ProductSchema` is updated, `OrderSchema` will automatically include `brand_id` in nested product objects — but if `ProductSchema` is left stale, order API responses will also be missing the field.

**Before (`backend/app/schemas/order.py`):**
```python
class OrderItemSchema(BaseModel):
    product: ProductSchema  # ProductSchema missing brand_id → propagates omission
    quantity: int
```

**After:** No change to `order.py` is required *if* `ProductSchema` is updated first. Verify the nested serialization is correct by inspecting the order API response after updating `ProductSchema`.

---

### 5. Discount and Order Service Logic Always Evaluates `brand_id` as `None`

Any brand-conditional logic in `discount.py` or `order_service.py` will silently fall through to the null/default path.

**Before (`backend/app/services/discount.py`):**
```python
def get_discount(product: Product) -> float:
    if product.brand_id == PREMIUM_BRAND_ID:  # Always False — brand_id never loaded
        return 0.15
    return 0.0
```

**After:** No code change needed to the conditional itself, but you must confirm the ORM session is loading `brand_id` correctly after the model is confirmed updated. Add an assertion in tests (see QA section).

## Required Actions by Role

### 🔧 Backend Developer

1. **`backend/app/schemas/product.py`** — Add `brand_id: Optional[int] = None` to `ProductSchema` and `ProductCreateSchema` / `ProductUpdateSchema` if they exist separately.
2. **`backend/app/routers/products.py`** — Update `create_product` and `update_product` endpoints to pass `brand_id` when constructing or updating the `Product` ORM object.
3. **`backend/app/schemas/order.py`** — Verify the nested `ProductSchema` in order responses correctly reflects `brand_id` after `ProductSchema` is updated. No structural change expected, but validate via integration test.
4. **`backend/app/services/order_service.py`** — Audit any access to `product.brand_id` within the order service. Ensure the ORM model is refreshed or eagerly loaded before access. Add explicit handling for the `None` case.
5. **`backend/app/services/discount.py`** — Audit brand-conditional discount logic. Confirm `product.brand_id` resolves to the correct value from the database. Test both `brand_id=None` and `brand_id=<valid_id>` paths explicitly.
6. **`backend/app/seed.py`** — Update seed data to populate `brand_id` with at least one realistic non-null value so brand features are testable in development and staging environments.
7. **`backend/app/models/order.py`** — If `Order` accesses `product.brand_id` for denormalized storage or eager loading, confirm the `Product` SQLAlchemy model is updated and re-run any affected migration scripts.
8. **`backend/app/database.py`** — Confirm the Alembic migration for the `brand_id` column has been generated and applied. Do not rely solely on `create_all()`.
9. **`backend/app/routers/orders.py`** / **`backend/app/routers/users.py`** / **`backend/app/routers/auth.py`** — Audit for any inline `Product` construction or serialization that bypasses `ProductSchema`.

### 🖥️ Frontend Developer

1. **`frontend/src/types/product.ts`** — Add `brand_id?: number | null` to the `Product` interface. This is the highest-priority frontend change; all downstream issues stem from this omission.
2. **`frontend/src/hooks/useProducts.ts`** — After updating the type, audit the hook's return type and any transformation logic (e.g., `map`, `filter`, object spread). Ensure `brand_id` is not dropped in intermediate data transformations.
3. **`frontend/src/pages/ProductsPage.tsx`** — Audit any place where `Product` objects are spread (`{ ...product }`), passed as props, or rendered. Add handling for the nullable case (`brand_id` may be `null` or `undefined`).
4. Globally search for `product.brand` or brand-related display components that may have been written speculatively and are now silently receiving `undefined`. Update to read `product.brand_id` with a null guard.

---

### 🧪 QA / Test Engineer

1. **`backend/tests/conftest.py`** — Update all `Product` fixture factory functions to include `brand_id`. Add at least two fixture variants: one with `brand_id=None` and one with a valid `brand_id` (e.g., `brand_id=1`).
2. **`backend/tests/test_orders.py`** — Update expected response shapes in order tests to include `brand_id` in nested product objects. Tests that assert on exact dict/JSON shapes will fail or give false positives without this update.
3. **`backend/tests/test_discount.py`** — Add explicit test cases for discount logic with `brand_id=None` (fallback path) and `brand_id=<premium>` (discount path). Confirm the discount calculation is not silently locked to the null branch.
4. Write a new integration test that:
   - POSTs a product with `brand_id=2`
   - GETs the product and asserts `brand_id == 2` in the response
   - Creates an order containing that product and asserts `brand_id` appears in the order response
5. Update **`docs/requirements.md`** and **`docs/scenarios.md`** to document the `brand_id` field, its nullability, and any brand-association business rules.

### Example 1: Full Pydantic Schema Update

```python
# backend/app/schemas/product.py
from typing import Optional
from pydantic import BaseModel

# BEFORE
class ProductBase(BaseModel):
    name: str
    price: float

class ProductCreateSchema(ProductBase):
    pass  # brand_id missing — will be stripped from POST body

class ProductSchema(ProductBase):
    id: int

    class Config:
        from_attributes = True


# AFTER
class ProductBase(BaseModel):
    name: str
    price: float
    brand_id: Optional[int] = None  # ← propagates to all subclasses

class ProductCreateSchema(ProductBase):
    pass  # brand_id now inherited ✓

class ProductUpdateSchema(ProductBase):
    name: Optional[str] = None
    price: Optional[float] = None
    # brand_id inherited as Optional[int] = None ✓

class ProductSchema(ProductBase):
    id: int

    class Config:
        from_attributes = True
```


```typescript
// frontend/src/hooks/useProducts.ts

// BEFORE — brand_id silently dropped in transformation
const useProducts = () => {
  const { data } = useQuery<Product[]>('/api/products');
  return data?.map(({ id, name, price }) => ({ id, name, price }));
  //                                          ↑ destructuring drops brand_id
};

// AFTER
const useProducts = () => {
  const { data } = useQuery<Product[]>('/api/products');
  return data?.map(({ id, name, price, brand_id }) => ({
    id,
    name,
    price,
    brand_id,  // ← explicitly carried through
  }));
};
```

### Example 3: Router Write Path

```python
# backend/app/routers/products.py

# BEFORE — brand_id in request body is silently ignored
@router.post("/products", response_model=ProductSchema, status_code=201)
async def create_product(
    product_in: ProductCreateSchema,
    db: Session = Depends(get_db),
):
    db_product = Product(
        name=product_in.name,
        price=product_in.price,
    )
    db.add(db_product)
    db.commit()
    db.refresh(db_product)
    return db_product


# AFTER — brand_id is persisted
@router.post("/products", response_model=ProductSchema, status_code=201)
async def create_product(
    product_in: ProductCreateSchema,
    db: Session = Depends(get_db),
):
    db_product = Product(
        name=product_in.name,
        price=product_in.price,
        brand_id=product_in.brand_id,  # ← persisted to DB
    )
    db.add(db_product)
    db.commit()
    db.refresh(db_product)
    return db_product
```
