# Migration Guide: `phone` Field Added to `User` Model

**Date:** 2026-06-28
**Severity:** Medium | **Scope:** Full-stack (Backend, Frontend, QA)

---

## Summary

A new optional `phone` column (`str | None`, nullable) has been added to the `User` database model. While the change is additive and non-destructive at the database level, it introduces a cascade of required updates across Pydantic schemas, API routers, TypeScript types, frontend components, seed data, and test fixtures. Without these updates, the field will be silently stripped from API responses, ignored on input, and will cause TypeScript compile errors or runtime `undefined` values in the frontend. All teams must act before this change is considered fully integrated.

---

## Breaking Changes

### 1. `User` Model — New Nullable Column

The `User` SQLAlchemy model now declares a `phone` column.

**Before:**
```python
# backend/app/models/user.py
class User(Base):
    __tablename__ = "users"

    id: Mapped[int] = mapped_column(primary_key=True)
    email: Mapped[str] = mapped_column(unique=True)
    hashed_password: Mapped[str]
```

**After:**
```python
# backend/app/models/user.py
class User(Base):
    __tablename__ = "users"

    id: Mapped[int] = mapped_column(primary_key=True)
    email: Mapped[str] = mapped_column(unique=True)
    hashed_password: Mapped[str]
    phone: Mapped[str | None] = mapped_column(nullable=True, default=None)
```

---

### 2. Pydantic Schemas — `phone` Must Be Declared Explicitly

Without this change, `phone` is **silently omitted** from all API responses and inputs.

**Before:**
```python
# backend/app/schemas/user.py (inferred — adjust to your actual schema file)
class UserRead(BaseModel):
    id: int
    email: str

class UserCreate(BaseModel):
    email: str
    password: str

class UserUpdate(BaseModel):
    email: str | None = None
```

**After:**
```python
# backend/app/schemas/user.py
class UserRead(BaseModel):
    id: int
    email: str
    phone: str | None = None          # ← ADD

class UserCreate(BaseModel):
    email: str
    password: str
    phone: str | None = None          # ← ADD

class UserUpdate(BaseModel):
    email: str | None = None
    phone: str | None = None          # ← ADD
```

---

### 3. TypeScript `User` Type — Missing Field Causes Compile Errors

**Before:**
```typescript
// frontend/src/types/user.ts
export interface User {
  id: number;
  email: string;
}
```

**After:**
```typescript
// frontend/src/types/user.ts
export interface User {
  id: number;
  email: string;
  phone: string | null;               // ← ADD
}
```

---

### 4. Auth Router — Registration/Login Responses Silently Omit `phone`

If the auth router uses `UserRead` (or an inline schema) as its response model, no code change is needed once the schema is updated per Breaking Change #2. However, verify that response models are explicitly declared:

**Before:**
```python
# backend/app/routers/auth.py
@router.post("/register")
async def register(data: UserCreate, db: Session = Depends(get_db)):
    user = create_user(db, data)
    return user  # ← No response_model; phone may leak or be missing
```

**After:**
```python
# backend/app/routers/auth.py
@router.post("/register", response_model=UserRead)   # ← Explicit model
async def register(data: UserCreate, db: Session = Depends(get_db)):
    user = create_user(db, data)
    return user
```

---

### 5. Test Fixtures — Shape Assertions Will Fail

**Before:**
```python
# backend/tests/conftest.py
@pytest.fixture
def sample_user():
    return {
        "id": 1,
        "email": "test@example.com",
    }
```

**After:**
```python
# backend/tests/conftest.py
@pytest.fixture
def sample_user():
    return {
        "id": 1,
        "email": "test@example.com",
        "phone": None,                # ← ADD (serialized API response now includes this)
    }
```

## Required Actions by Role

### 🔧 Backend Developer

1. **Run the database migration.** Generate and apply an Alembic migration for the new nullable `phone` column:
   ```bash
   alembic revision --autogenerate -m "add_phone_to_users"
   alembic upgrade head
   ```
   Verify the generated migration adds `nullable=True` and no `server_default` is required (nullable columns accept `NULL` by default).

2. **Update all User Pydantic schemas** (`UserRead`, `UserCreate`, `UserUpdate`) to include `phone: str | None = None`.

3. **Verify routers** (`backend/app/routers/users.py`, `backend/app/routers/auth.py`) use explicit `response_model=UserRead` on all User-returning endpoints.

4. **Update `seed.py`** to optionally populate `phone` on seeded User records (even `None` is acceptable, but consider adding a sample value for demo completeness).

5. **Trigger schema rebuild** if using `model_rebuild()` in `backend/app/models/__init__.py` or anywhere Pydantic resolves forward references:
   ```python
   UserRead.model_rebuild()
   ```

6. **Check `backend/app/common/deps.py`** — if `get_current_user` or similar dependency returns a typed `User` object used downstream, ensure nothing asserts a fixed field set that now needs updating.

---

### 🖥️ Frontend Developer

1. **Update `frontend/src/types/user.ts`** — add `phone: string | null` to the `User` interface (see Breaking Change #3).

2. **Update `useUsers.ts`** — if the hook maps or destructures User fields explicitly, add `phone` to any mapping logic:
   ```typescript
   // frontend/src/hooks/useUsers.ts
   const mapUser = (raw: any): User => ({
     id: raw.id,
     email: raw.email,
     phone: raw.phone ?? null,        // ← ADD
   });
   ```

3. **Update `UsersPage.tsx`** — decide whether to display `phone` in the UI:
   ```tsx
   // frontend/src/pages/UsersPage.tsx
   <td>{user.phone ?? '—'}</td>       // ← ADD or handle gracefully
   ```
   If `phone` is not yet displayed, still ensure the `User` type is updated so TypeScript does not error when the field is accessed elsewhere.

4. **Run TypeScript compiler** to catch any residual type errors:
   ```bash
   npx tsc --noEmit
   ```

---

### 🧪 QA Engineer

1. **Update fixture dicts** in `backend/tests/conftest.py` to include `"phone": None` in any User dict used in response shape assertions.

2. **Review `backend/tests/test_orders.py`** and **`backend/tests/test_discount.py`** — if these tests create User objects directly or assert full response payloads, update the expected shapes to include `phone: null`.

3. **Add new test cases** for the `phone` field:
   - `POST /register` with `phone` supplied → verify it appears in the response.
   - `POST /register` without `phone` → verify response contains `"phone": null`.
   - `PATCH /users/{id}` with a valid phone string → verify update persists.
   - `PATCH /users/{id}` with `phone: null` → verify explicit null clears the field.

4. **Verify OpenAPI spec** reflects the new field at `/docs` or `/openapi.json` after the server restarts. The `phone` field should appear in `UserRead`, `UserCreate`, and `UserUpdate` schema components.

5. **Regression-test auth flows** — confirm that `/register` and `/login` (or `/me`) responses now include `"phone": null` for existing users.

#### `backend/app/routers/users.py`

```python
# BEFORE
@router.patch("/{user_id}", response_model=UserRead)
async def update_user(
    user_id: int,
    data: UserUpdate,
    db: Session = Depends(get_db),
):
    user = db.get(User, user_id)
    user.email = data.email or user.email
    db.commit()
    return user


# AFTER
@router.patch("/{user_id}", response_model=UserRead)
async def update_user(
    user_id: int,
    data: UserUpdate,
    db: Session = Depends(get_db),
):
    user = db.get(User, user_id)
    user.email = data.email or user.email
    if data.phone is not None or "phone" in data.model_fields_set:
        user.phone = data.phone       # ← ADD: handle explicit null-setting
    db.commit()
    return user
```
